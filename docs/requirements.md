# Food Roulette 要件定義 v0.1

更新日: 2026-04-01
対象: ポートフォリオ用 Web アプリ（個人利用）

## 1. 背景と目的

- 優柔不断なときに、食事の意思決定を素早くする。
- 用途の異なる 2 種類のルーレットを提供する。
  - 1) 晩御飯ルーレット（料理候補を決める）
  - 2) お店ルーレット（外食時の店舗候補を決める）
- ルーレット利用はログイン不要、データメンテナンスは管理者ログイン必須にする。

## 2. スコープ

### 2.1 対象（In Scope）

- 一般ユーザー向け
  - 2種類のルーレット画面をログインなしで利用できる
  - 条件を選択して「気に入るまで繰り返し回せる」
- 管理者向け
  - ログイン機能（ID/Password）
  - 料理データ・店舗データの手入力登録/編集/削除
- データ
  - 料理データと店舗データは完全に分離
  - 店舗エリアは初期値として「なんば」「梅田」を登録可能
  - 将来的にエリア追加できる設計

### 2.2 対象外（Out of Scope: 初版では実装しない）

- 履歴保存
- お気に入り保存
- 重複排除ロジック（同じ候補を出さない制御）
- 外部サービスからの自動データ取得（API連携、スクレイピング）
- CSV一括インポート

## 3. 利用者と権限

- 一般ユーザー（匿名）
  - ルーレット利用のみ可能
  - データ編集不可
- 管理者（あなた）
  - 認証後、管理画面でデータCRUD可能

## 4. 機能要件

## 4.1 共通要件

- 2つのルーレットはトップから遷移可能
- 条件を指定して実行できる
- 「もう一回回す」で再抽選できる
- 候補が0件の場合は、条件緩和を促すメッセージを表示する

### 4.2 晩御飯ルーレット（Roulette 1）

#### 4.2.1 目的

- 今日作る/食べる料理を決める

#### 4.2.2 候補データ粒度（提案）

- 1レコード = 1料理（例: 親子丼、麻婆豆腐）
- 料理ごとにタグを持たせ、条件抽出後にルーレット抽選

#### 4.2.3 絞り込み条件（提案）

レシピサイトの一般的な絞り込みを参考に、初版は以下を推奨:

- ジャンル（和食/洋食/中華/韓国/エスニック/その他）
- 料理区分（主菜/副菜/汁物/主食/デザート）
- 調理時間帯（~10分 / 11-20分 / 21-30分 / 31分~）
- 予算帯（~300円 / 301-600円 / 601-1000円 / 1001円~）
- シーンタグ（がっつり/さっぱり/温かい/冷たい）
- 任意フリーワード（料理名部分一致）

初版での入力負荷を下げるため、必須項目は「料理名・ジャンル・料理区分」までに抑える。

### 4.3 お店ルーレット（Roulette 2）

#### 4.3.1 目的

- 友人との外食時に行く店を決める

#### 4.3.2 絞り込み条件（提案）

飲食店検索サイトの一般的な条件を参考に、初版は以下を推奨:

- エリア（初期: なんば / 梅田）
- ジャンル（居酒屋/焼肉/寿司/イタリアン/ラーメン/カフェ/その他）
- 予算帯（ランチ・ディナーどちらか1軸で開始）
- 営業時間帯タグ（ランチ営業あり/ディナー営業のみ/深夜可）
- 席・利用条件タグ（禁煙/喫煙可、個室あり など）
- 任意フリーワード（店名部分一致）

初版では、入力負荷とのバランスを考え「エリア・ジャンル・予算帯」を主要条件にする。

### 4.4 管理画面

- ログイン/ログアウト
- 料理データ管理
  - 一覧、追加、編集、削除
- 店舗データ管理
  - 一覧、追加、編集、削除
- マスタ管理（任意）
  - エリア、ジャンル、タグの追加・編集

## 5. データ要件（論理）

### 5.1 料理データ（例）

- id
- name（料理名）
- genre
- course_type（主菜/副菜など）
- cook_time_min
- budget_yen
- tags（複数）
- is_active（表示対象フラグ）
- created_at / updated_at

### 5.2 店舗データ（例）

- id
- name（店名）
- area（なんば/梅田/将来追加）
- genre
- budget_range
- open_tags（ランチ、深夜など）
- facility_tags（個室、喫煙可など）
- note（メモ）
- is_active
- created_at / updated_at

### 5.3 管理者データ

- id
- email（または login_id）
- password_hash
- role（admin 固定でも可）
- created_at / updated_at

## 6. 非機能要件

- コスト
  - 運用費は極小（無料枠中心）
- 可用性
  - ポートフォリオ閲覧時に安定稼働すること
- 性能
  - 1ユーザー利用で体感遅延がないこと（抽選は即時）
- セキュリティ
  - 管理画面は認証必須
  - パスワード平文保存禁止（ハッシュ化）
- 保守性
  - 料理/店舗の2データを分離し、今後の項目追加に対応しやすい構造

## 7. 技術・デプロイ方針（低コスト案）

以下は 2026-04 時点の公開情報ベース。最終選定時に最新料金を再確認する。

### 案A（最有力・実装容易）

- Frontend: Vercel Hobby（無料）
- Backend/DB/Auth: Supabase Free（Postgres + Auth）
- 特徴:
  - 開発速度が速い
  - 管理画面ログインも実装しやすい
  - SQLベースで拡張しやすい
- 注意:
  - 無料プロジェクトの休眠停止条件あり

### 案B（Google系でまとまる）

- Frontend Hosting + Backend: Firebase（Hosting + Firestore + Auth）
- 特徴:
  - 学習資料が多い
  - 小規模なら無料枠で収まりやすい
- 注意:
  - 機能によっては Blaze 必須になる領域があるため構成注意

### 案C（無料枠重視の軽量構成）

- Frontend: Cloudflare Pages or Vercel
- API/DB: Cloudflare Workers + D1
- 特徴:
  - 無料枠が強く、軽量アプリ向き
  - エッジ実行でレスポンスが速い
- 注意:
  - 実装難易度は案Aよりやや高め

推奨は、ポートフォリオとしての開発速度と見栄えのバランスから案A。

## 8. 受け入れ基準（初版）

- ログイン不要で 2つのルーレットを利用できる
- 条件指定して抽選できる
- 管理者ログイン後に料理/店舗データをCRUDできる
- 料理と店舗で別データを参照している
- 店舗エリアとして「なんば」「梅田」を扱える

## 9. 今後の拡張候補

- 抽選済み候補の一時除外（重複回避モード）
- お気に入り登録
- 履歴表示
- CSVインポート
- 地図リンク、営業時間自動更新連携
